#### 문제 1: 테이블 생성하기 (CREATE TABLE)

1. `attendance` 테이블은 중복된 데이터가 쌓이는 구조이다. 중복된 데이터는 어떤 컬럼인가?

> crew_id

2. `attendance` 테이블에서 중복을 제거하기 위해 `crew` 테이블을 만들려고 한다. 어떻게 구성해 볼 수 있을까?
   > crew 테이블이름
   > crew_id(PK), nickname을 속성으로 가지는 테이블을 생성하고
   > 기준 attendance에 nickname 컬럼을 제거 및 crew_id를 FK로 등록한다.
3. `crew` 테이블에 들어가야 할 크루들의 정보는 어떻게 추출할까? (hint: DISTINCT)

```sql
SELECT DISTINCT crew_id, nickname
FROM attendance;
```

3. 최종적으로 crew 테이블 생성:

```sql
CREATE TABLE crew (
crew_id INT NOT NULL AUTO_INCREMENT,
nickname VARCHAR(50) NOT NULL,
PRIMARY KEY (crew_id)
;
```

5. attendance 테이블에서 크루 정보를 추출해서 crew 테이블에 삽입하기:

```sql
INSERT INTO crew (crew_id, nickname)
SELECT DISTINCT crew_id, nickname
FROM attendance;
```

#### 문제 2: 테이블 컬럼 삭제하기 (ALTER TABLE)

1. crew 테이블을 만들고 중복을 제거했다. attendance에서 불필요해지는 컬럼은?

> nickname

2. 컬럼을 삭제하려면 어떻게 해야 하는가?

```sql
ALTER TABLE attendance
DROP COLUMN nickname;
```

#### 문제 3: 외래키 설정하기

`attendance`에서 관심사의 분리를 통해 `crew` 테이블을 별도로 만들었다. 따라서, 나중에 nickname이 필요하다면 `crew` 테이블에서 확인하면 된다.

그런데 잠재적인 문제가 남아 있다:

- 만약에 `crew` 테이블에는 `crew_id`가 `12`번인 크루가 존재하지 않지만, `attendance` 테이블에는 여전히 `crew_id`가 `12`번인 크루가 존재한다면?
  - 해당 크루가 중간에 퇴소했거나
  - 누군가의 실수에 의해 레코드가 삭제되었거나

```sql
ALTER TABLE attendance
ADD CONSTRAINT fk_attendance_crew
FOREIGN KEY (crew_id) REFERENCES crew(crew_id);
```


```sql
SELECT c.nickname FROM attendance AS a
JOIN crew AS c ON a.crew_id = c.crew_id
WHERE attendance_date = '2025-03-05'
ORDER BY end_time DESC
LIMIT 1

-- 공동 1등 문제 개선
SELECT c.nickname, a.end_time
FROM attendance AS a
JOIN crew AS c ON a.crew_id = c.crew_id

  AND a.end_time = (
      SELECT MAX(end_time)
      FROM attendance
      WHERE attendance_date = '2025-03-05'
  );
```

#### 문제 13: 크루별로 '기록된' 날짜 수 조회

```sql
SELECT a.crew_id, c.nickname, COUNT(*) AS dc
FROM attendance AS a
JOIN crew AS c ON a.crew_id = c.crew_id
GROUP BY a.crew_id
ORDER BY dc DESC, crew_id
;

-- 개선

SELECT a.crew_id, c.nickname, COUNT(DISTINCT a.attendance_date) AS dc
FROM attendance AS a
JOIN crew AS c ON a.crew_id = c.crew_id
GROUP BY a.crew_id
ORDER BY dc DESC, crew_id;
```

#### 문제 14: 크루별로 등교 기록이 있는(start_time IS NOT NULL) 날짜 수 조회

```sql
SELECT c.nickname, count(DISTINCT a.attendance_date) AS date_count
FROM attendance AS a
JOIN crew as c ON a.crew_id = c.crew_id
WHERE start_time IS NOT NULL
GROUP BY a.crew_id, c.nickname
ORDER BY date_count DESC
;
```

#### 문제 15: 날짜별로 등교한 크루 수 조회

```sql
SELECT a.attendance_date, count(*) AS crew_count
FROM attendance AS a
JOIN crew as c ON a.crew_id = c.crew_id
GROUP BY attendance_date
ORDER BY a.attendance_date
;
```

#### 문제 16: 크루별 가장 빠른 등교 시각(MIN)과 가장 늦은 등교 시각(MAX)

```sql
SELECT a.attendance_date, a.start_time, c.nickname
FROM attendance AS a
JOIN crew AS c ON a.crew_id = c.crew_id
WHERE (a.attendance_date, a.start_time) IN (
    SELECT attendance_date, MIN(start_time)
    FROM attendance
    GROUP BY attendance_date
)
OR (a.attendance_date, a.start_time) IN (
    SELECT attendance_date, MAX(start_time)
    FROM attendance
    GROUP BY attendance_date
)
ORDER BY a.attendance_date, a.start_time;
```

추가 조언: '주소'라는 표현도 좋지만, 현실 세계의 **'주민등록번호'**에 비유하면 더 완벽합니다. 주소는 이사하면 바뀔 수 있지만, 기본키는 한 번 정해지면 절대 변하지 않아야(Immutable) 하기 때문입니다.

왜 필요한가?: 데이터의 **'유일성(Uniqueness)'**과 **'최소성(Minimality)'**을 보장합니다. PK가 없다면 이름이 같은 두 명의 '검프' 중 누가 누구인지 구분할 방법이 없어 데이터가 오염됩니다.

- 출제 의도: 테이블에서 각 레코드를 고유하게 식별하는 기본키의 개념과 중요성을 이해해보자. 실제 데이터를 다룰 때 식별자가 없다면 어떤 문제가 발생할지 생각해보고, 기본키 선택이 데이터베이스 설계에 미치는 영향을 고민해보자.

2. **MySQL에서 사용되는 `AUTO_INCREMENT`는 왜 필요할까?**

**내 생각**

> 기본키의 유일성 보장 및 중복 방지
> 식별만 할 수 있으면 되는 의미가 없는 숫자이므로 자동 기입 기능으로 편의를 제공

**AI 의견**

추가 조언: 이를 **인공 키(Surrogate Key)**라고 부릅니다. 비즈니스적 의미(닉네임, 이메일 등)가 담긴 값을 PK로 쓰면 나중에 그 정보가 바뀔 때 연관된 모든 테이블을 고쳐야 하는 대참사가 일어납니다. AUTO_INCREMENT는 이런 위험을 방지하고 시스템이 관리하기 가장 편한 '순수한 번호'를 부여해 줍니다.

- 출제 의도: 일일이 ID 값을 지정해야 하는 번거로움을 줄이고 중복 없는 고유값을 자동으로 생성하는 기능의 필요성을 파악해보자.

- 출제 의도: NULL 처리는 SQL 학습에서 자주 혼란을 주는 개념이다. 특히 프론트엔드에서 NULL 데이터를 어떻게 표시할지는 실무적으로 중요한 문제이다.

5. **출석 시스템에서 동시에 100명이 등교 버튼을 누른다면 어떤 일이 일어날까? 이 문제를 [2026 공통강의 - DB](https://techcourse.woowahan.com/s/uLX8Ma07/ls/62meRAXd)에서 배운 트랜잭션과 ACID 속성으로 설명해보자.**

**내 생각**

> 클라이언트가 서버에게 요청을 보낸다.
>
> 1. 예를 들면 출석이라는 행위를 호출한다.
> 2. 자동 커밋 옵션을 끈다.
> 3. 출석은 커넥션을 얻어서 자신의 기본키를 이용해서 조회하고 출석 데이터를 INSERT/UPDATE한다.
> 4. 커넥션을 반납한다.
>    2번을 진행 중 예외 발생 시 롤백을 한다.
>    원자성: 출석이라는 트랜잭션이 실패한다면 적용한 DML도 취소되어야 한다.
>    지속성: 출석을 안한 인원 + 출석을 한 인원의 합이 계속 일관되어야 한다.
>    독립성: 해당 테이블에 DML을 할 때에 동시성 제어를 해야 한다.
>    영구성: 복제본이 존재해서 문제가 발생해도 정상적으로 사용 가능해야 한다.

**AI 의견**

1. 원자성 (Atomicity): "전부 아니면 전무"
   상황: 100명 중 한 명의 요청이 처리되던 중, 서버가 꺼지거나 DB 에러가 발생합니다.
   역할: 등교 데이터는 INSERT 되었는데, 크루의 포인트 점수는 UPDATE 되지 않는 등의 '중간 단계'가 남아서는 안 됩니다. 성공해서 다 반영되거나, 실패해서 아예 없었던 일이 되거나 둘 중 하나여야 합니다.
2. 일관성 (Consistency): "룰은 깨지지 않는다"
   상황: 우리 시스템에는 '한 명의 크루는 같은 날에 두 번 등교할 수 없다'는 UNIQUE 제약 조건이 있습니다.
   역할: 100명이 동시에 눌러도, DB는 미리 정해진 무결성 제약 조건을 위반하는 데이터는 절대 허용하지 않습니다. 트랜잭션 전후의 데이터베이스는 항상 '유효한 상태'여야 합니다.
3. 격리성 (Isolation): "너는 너, 나는 나"
   상황: 100명의 트랜잭션이 한꺼번에 몰려듭니다.
   역할: 각 트랜잭션은 마치 자기 혼자 DB를 쓰고 있는 것처럼 느껴야 합니다.
   이를 위해 DB는 **잠금(Lock)**을 사용합니다. 한 유저의 데이터를 수정하는 동안 다른 유저가 그 데이터를 건드리지 못하게 줄을 세우는 것이죠.
   격리 수준(Isolation Level)에 따라 동시 처리 속도와 데이터 정확도 사이의 트레이드오프가 발생합니다.
4. 지속성 (Durability): "기억은 사라지지 않는다"
   상황: 100명의 등교 처리가 '커밋 완료' 되자마자 캠퍼스 전체가 정전되었습니다.
   역할: 커밋이 완료된 데이터는 **비휘발성 저장소(디스크)**에 안전하게 기록되어야 합니다. 서버를 다시 켰을 때, 정전 직전에 완료된 100명의 출석 기록은 반드시 남아있어야 합니다. (로그 파일 등을 통해 복구하는 기능이 포함됩니다.)

🛠 100명 동시 요청 시 발생하는 실무적 문제
단순히 ACID를 지키는 것을 넘어, 실제로 100명이 동시에 버튼을 누르면 DB 엔진 내부에서는 이런 일이 벌어집니다.
커넥션 풀(Connection Pool) 고갈: 서버가 가질 수 있는 커넥션은 한정되어 있습니다. 100명이 동시에 커넥션을 점유하면 뒤에 오는 요청은 대기 상태가 됩니다.
데드락(Deadlock) 위험: 유저 A와 유저 B가 서로의 자원(Lock)이 풀리기를 기다리며 무한 대기에 빠질 수 있습니다.
행 잠금(Row Lock): 같은 크루가 여러 기기에서 동시에 누르는 게 아니라면 각기 다른 행을 INSERT 하므로 충돌이 적지만, 만약 '전체 출석 인원'을 카운트하는 특정 행을 매번 UPDATE 한다면 엄청난 병목 현상이 발생할 것입니다.

- 출제 의도: 실습에서 직접 다룬 INSERT/UPDATE가 실제 운영 환경에서는 동시성 문제를 일으킬 수 있다. 원자성(Atomicity)과 격리성(Isolation)이 왜 필요한지 출석 시스템의 맥락에서 구체적으로 떠올려보자.

6. **출석 데이터가 파일(CSV)이 아닌 데이터베이스에 저장되는 이유는 무엇일까? 파일 시스템으로 출석을 관리했다면 어떤 문제가 생길까?**

**내 생각**

> 각각의 로컬에 종속적이게 됨
> 그러므로 중복 데이터가 발생
> 같은 데이터를 공유하게 될 때 동시성 문제가 생길 수 있다.
> 트랜잭션 기능을 구현이 어렵다.

**AI 의견**

1. 데이터 중복과 불일치 (Data Redundancy & Inconsistency)
   파일 시스템: 여러 사람이 각자의 로컬에서 CSV를 관리하면, '어셔'의 닉네임이 한 곳에서는 '어셔', 다른 곳에서는 'Usher'로 저장될 수 있습니다. 데이터의 **단일 진실 공급원(Single Source of Truth)**이 무너집니다.
   DB: 정규화를 통해 데이터를 한 곳에서 관리하며, 유니크 제약 조건 등으로 중복과 불일치를 원천 차단합니다.
2. 동시 제어 불능 (Concurrency Issue)
   파일 시스템: 두 명의 코치가 동시에 하나의 CSV 파일을 열어 수정을 시도하면, 나중에 저장한 사람의 데이터가 먼저 저장한 사람의 데이터를 덮어씌우는 Lost Update 현상이 발생합니다. 파일 단위의 Lock은 너무 무거워서 효율적인 공유가 불가능합니다.
   DB: 레코드(행) 단위의 미세한 잠금(Lock)을 통해 100명이 동시에 버튼을 눌러도 데이터가 꼬이지 않게 관리합니다.
3. 트랜잭션과 무결성 보장 (Transaction & Integrity)
   파일 시스템: 파일에 데이터를 쓰던 중 정전이 나면 파일 자체가 깨지거나 데이터가 반만 기록될 수 있습니다. 이를 복구하는 로직을 매번 직접 구현해야 합니다.
   DB: ACID 특성을 통해 "전부 성공하거나, 아니면 아예 수행되지 않은 상태"를 보장합니다. 시스템 장애 시에도 로그를 통해 완벽한 복구가 가능합니다.
4. 보안 및 접근 제어 (Security)
   파일 시스템: CSV 파일은 운영체제의 권한 관리(읽기/쓰기)에 의존합니다. 파일에 접근 권한이 있는 사람은 모든 출석 데이터를 다 볼 수 있고, 특정 컬럼(예: 개인정보)만 가리는 세밀한 제어가 어렵습니다.
   DB: 사용자별로 테이블, 행, 심지어 컬럼 단위까지 접근 권한을 설정할 수 있습니다.

## 🧐 더 생각해 보기 (심화)

1. **왜 crew 테이블에서 nickname을 기본키로 하지 않은 걸까? attendance 테이블에 attendance_id가 존재하는 이유는 무엇일까?**

**내 생각**

> 기본키는 유일해야 하고, 변경이 최소화 되어야 한다.
> 닉네임은 중복이 발생할 수 있고 변경 가능성이 상대적으로 더 크다.

**AI 의견**

1. 왜 nickname을 기본키(PK)로 쓰지 않는가?
   닉네임처럼 비즈니스적 의미를 가진 데이터를 **자연키(Natural Key)**라고 합니다. 이를 PK로 쓰지 않는 이유는 크게 세 가지입니다.

불변성(Immutability) 부족: PK의 제1원칙은 "절대 변하지 않아야 한다"입니다. 닉네임은 사용자의 변심으로 언제든 바뀔 수 있습니다. 만약 닉네임을 PK로 썼는데 이를 수정하면, 그 닉네임을 외래키(FK)로 참조하던 수만 개의 출석 데이터까지 모두 수정해야 하는 대작업이 발생합니다.

유일성(Uniqueness)의 위험: 현재는 중복을 금지하더라도, 서비스 규모가 커지거나 정책이 바뀌어 동명이인을 허용해야 하는 순간이 오면 데이터베이스 구조 전체를 갈아엎어야 합니다.

성능 저하: 인덱싱과 조인(Join) 시 문자열(VARCHAR)은 정수(INT)보다 비교 연산 속도가 느리고 저장 공간을 더 많이 차지합니다.

2. 왜 attendance_id가 존재하는가? (대리키의 필요성)
   출석 테이블의 attendance_id는 아무런 비즈니스 의미가 없는 **대리키(Surrogate Key)**입니다.

정교한 레코드 식별: 한 명의 크루(crew_id)가 여러 날짜(attendance_date)에 걸쳐 데이터를 쌓습니다. 만약 특정 날짜의 출석 기록 하나가 잘못되어 수정하거나 삭제해야 할 때, attendance_id라는 고유 번호가 있으면 단 한 줄의 쿼리로 정확하게 타격할 수 있습니다.

복합키의 복잡성 회피: attendance_id가 없다면 crew_id와 attendance_date를 묶어서 PK로 써야 합니다(복합키). 하지만 복합키는 조인할 때마다 두 개 이상의 컬럼을 대조해야 하므로 쿼리가 복잡해지고 성능에 불리합니다.

시스템 일관성: 모든 테이블에 일관되게 id 형태의 대리키를 두면, ORM(JPA 등) 라이브러리를 사용하거나 공통화된 로직을 처리할 때 개발 생산성이 비약적으로 향상됩니다.

데이터베이스 설계에서 **"의미가 있는 값(닉네임, 날짜 등)"**은 데이터일 뿐, **"데이터를 가리키는 손가락(PK)"**이 되기엔 너무 불안정합니다. 그래서 우리는 변하지 않는 숫자(id)를 만들어 안전하게 데이터를 관리합니다.

- 출제 의도: 자연키(nickname)와 대리키(crew_id, attendance_id)의 차이점과 선택 기준을 이해해보자. 업무적 의미가 있는 데이터(nickname)는 미래에 변경될 가능성이 있어 기본키로 적합하지 않을 수 있다.

2. **데이터베이스 제약 조건 중 RESTRICT, CASCADE는 무엇인가?**

**내 생각**

> RESTRICT:
> CASCADE: 의존성이 연결된 부분을 하나로 취급

**AI 의견**

외래키(FK) 제약 조건에서 RESTRICT와 CASCADE는 부모 데이터가 바뀌거나 사라질 때, 이를 참조하는 자식 데이터를 어떻게 처리할 것인가에 대한 약속입니다.

1. RESTRICT (엄격한 거부)
   개념: 자식 테이블에 해당 데이터를 참조하는 행이 하나라도 있다면, 부모 테이블의 데이터를 삭제하거나 수정하는 것을 아예 막아버리는 설정입니다.
   작동 방식: crew 테이블에서 '어셔'를 삭제하려는데 attendance에 어셔의 기록이 남아있다면? DB가 "먼저 출석 기록부터 다 지우고 오세요!"라며 에러를 발생시킵니다.
   용도: 실수로 중요한 부모 데이터를 지워 데이터 정합성이 깨지는 것을 방지하고 싶을 때 사용합니다. (기본값인 경우가 많습니다.)
2. CASCADE (종속적 전파)
   개념: 부모 테이블의 데이터가 삭제되거나 수정되면, 이를 참조하는 자식 테이블의 데이터도 자동으로 함께 삭제되거나 수정되는 설정입니다.
   작동 방식: crew 테이블에서 '어셔'를 삭제하면? DB가 알아서 attendance 테이블을 뒤져 어셔의 모든 출석 기록을 조용히 같이 지워버립니다.
   용도: 연관된 데이터를 일일이 지우기 귀찮거나, 부모가 없는 자식 데이터는 존재 의미가 없을 때 사용합니다. (예: 게시글이 삭제되면 댓글도 자동 삭제)
   기록이 중요한 경우 (RESTRICT): 금융 거래, 수강 신청 기록 등은 함부로 지워지면 안 됩니다. 이럴 땐 RESTRICT로 설정해 사람이 직접 확인하고 처리하게 합니다.
   편의성이 중요한 경우 (CASCADE): "장바구니"나 "좋아요"처럼 부모(상품, 게시글)가 사라지면 의미가 없어지는 부가 데이터들은 CASCADE로 설정해 시스템이 알아서 청소하게 둡니다.

- 출제 의도: 외래키 관계에서 참조 무결성을 유지하기 위한 다양한 전략을 이해해보자. 예를 들어 사용자가 탈퇴할 때 그 사용자의 게시글도 함께 삭제해야 할지, 아니면 유지해야 할지와 같은 실제 의사결정에 이 개념이 어떻게 적용되는지 고민해보자.

3. **다음 두 쿼리는 동일한 결과를 반환하지만 성능에 차이가 있다. 어떤 차이가 있으며, 어떤 상황에서 각각 유리할까?**

**내 생각**

> 쿼리1: 크루 테이블의 모든 레코드를 순회하면서 LIKE 체크한다.
> 출석테이블의 개별 레코드마다 LIKE 통과된 레코드들과 비교를 하고 일치하는 경우 얼리 리턴된다.
> 쿼리2: 출석테이블의 개별 레코드마다 크루테이블의 레코드의 crew_id를 비교하는데 얼리 리턴안된다.
> 조인된 모든 레코드들을 LIKE 비교한다.

**AI 의견**

| **항목**        | **쿼리 1 (IN / 서브쿼리)**                           | **쿼리 2 (JOIN)**                                |
| --------------- | ---------------------------------------------------- | ------------------------------------------------ |
| **핵심 철학**   | **필터링(Filtering)** - 조건에 맞나?                 | **결합(Combining)** - 데이터 합치기              |
| **중복 처리**   | 서브쿼리에 같은 ID가 여러 개 있어도 결과에 영향 없음 | 자칫 1:N 관계에서 결과 행이 뻥튀기될 위험이 있음 |
| **인덱스 활용** | 메인 테이블의 인덱스를 주로 활용                     | 두 테이블의 인덱스를 모두 적절히 활용 가능       |
| **가독성**      | "이 명단에 있는 사람들 것만 가져와" (직관적)         | "출석부와 명단을 합쳐서 보여줘" (구조적)         |

```sql
-- 쿼리 1: 서브쿼리 사용
SELECT * FROM attendance WHERE crew_id IN (SELECT crew_id FROM crew WHERE nickname LIKE '네%');

-- 쿼리 2: JOIN 사용
SELECT a.* FROM attendance a JOIN crew c ON a.crew_id = c.crew_id WHERE c.nickname LIKE '네%';

```

- 출제 의도: 동일한 결과를 위한 다양한 접근법의 장단점을 이해하는 것은 실무에서 중요한 의사결정 역량이기도 하다.

4. **attendance 테이블을 완전히 정규화하면 어떤 장점이 있을까? 반대로 일부 비정규화를 적용한다면 어떤 쿼리 성능 이점을 얻을 수 있을까?**

**내 생각**

> 비정규화 시: 조인 비용과 작아져 조회가 빠르다, 수정 시 중복된 곳 모두 수정해야 하기때문에 느리다.

**AI 의견**

| **구분**        | **완전 정규화 (Standard)**    | **비정규화 (Optimization)**      |
| --------------- | ----------------------------- | -------------------------------- |
| **조회(Read)**  | **느림** (JOIN 연산 필요)     | **빠름** (단일 테이블 스캔)      |
| **쓰기(CUD)**   | **빠름** (한 곳만 수정)       | **느림** (중복된 곳 모두 수정)   |
| **무결성**      | 매우 높음 (중복 없음)         | 낮음 (데이터 꼬일 위험)          |
| **유리한 상황** | 데이터 정합성이 생명인 시스템 | 대규모 트래픽의 읽기 위주 서비스 |

- 출제 의도: 정규화와 성능 사이의 균형은 데이터베이스 설계의 핵심 과제이다.

6. **실습에서 수행한 INSERT, UPDATE, DELETE를 하나의 트랜잭션으로 묶는다면 어떻게 작성할 수 있을까? 만약 DELETE 도중 오류가 발생하면 앞서 수행한 INSERT와 UPDATE는 어떻게 되어야 할까?**

**내 생각**

> 만약 DELETE 도중 오류가 발새한다면 앞서 수행한 DML은 모두 롤백 되어야 한다.
> 왜냐하면 All or Nothing이기 때문이다.

**AI 의견**

작업이 취소되는 과정
자동 롤백: DELETE 도중 오류가 발생하면 데이터베이스 엔진은 해당 트랜잭션 내에서 수행된 모든 변경 사항을 무효화합니다.
이전 상태 복구: 앞서 성공했던 INSERT와 UPDATE의 결과물은 실제 디스크에 영구 반영되지 않고 메모리(또는 언두 로그) 상에만 존재하다가, 오류 즉시 명령 수행 전의 상태로 되돌아갑니다.
결과: 사용자 입장에서는 INSERT도 안 된 것이고, UPDATE로 올라갔던 숫자도 다시 원복된 것처럼 보이게 됩니다. 3. 실무적 조언: 에러 핸들링
애플리케이션(Java, Python 등)에서 DB를 다룰 때는 보통 try-catch 블록을 사용하여 이 로직을 구현합니다.
Try: 모든 DML을 순차적으로 실행.
Catch: 중간에 하나라도 에러가 나면 ROLLBACK 명령을 전송.
Finally: 성공 시에만 COMMIT 전송.
